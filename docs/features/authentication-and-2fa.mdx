Skyvern has full authentication support built in — passwords, 2FA, magic links, and password manager integrations. Secrets live in your vault (Bitwarden, 1Password, Azure Key Vault, or your own service); Skyvern never persists them and never sends them to the LLM.

---

## What's supported

**Credential types**
- Passwords (with optional TOTP secret bundled in)
- Credit cards (for checkout automations)
- Arbitrary secrets / API keys

**2FA methods**
- **TOTP / authenticator apps** — store the secret once; Skyvern generates valid codes on every run
- **Email or SMS codes** — push codes to Skyvern via API, or expose an endpoint Skyvern polls every 10 seconds
- **Magic links** — push the one-time login link; Skyvern detects URLs automatically and follows them
- **Multi-field TOTP** — Skyvern recognizes split-box layouts (one digit per input) and fills all fields with the same code

**Vault integrations**
- Bitwarden (Cloud & self-hosted / Vaultwarden)
- 1Password (via service account token)
- Azure Key Vault
- Custom HTTP vault — bring your own service

---

## Security by design

When a run needs a credential, Skyvern fetches it from your vault, injects it directly into the browser's form field, and discards it as soon as the field is filled. The value never enters an LLM prompt, never lands in a log line, and never gets written back to any persistent Skyvern storage.

This has a visible consequence: in your run recordings, action timelines, and LLM prompts you'll see stable placeholder tokens like `BW_PASSWORD`, `BW_USERNAME`, and `BW_TOTP` wherever real credentials were used. The prefix reflects the vault source (`BW_` for Bitwarden, `OP_` for 1Password, `AZ_` for Azure Key Vault, `SK_` for Skyvern's built-in vault) so you can audit which credential fired without the secret ever leaving your vault.
