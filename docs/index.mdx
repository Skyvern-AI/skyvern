<div className="fm-section" id="how-it-works">
  <div className="fm-section-head">
    <span className="fm-section-num">01</span>
    <h2 className="fm-section-title">How it works</h2>
  </div>

  <p className="fm-prose">
    Every Skyvern automation, whether you trigger it from the dashboard, the API, or a workflow integration, runs the same five-step loop. Each cycle through this loop is called a <strong>step</strong>, and a typical task completes in 2 to 10 steps. Because Skyvern reads the page visually instead of relying on hardcoded selectors, it works on sites it has never seen before and keeps working when those sites redesign.
  </p>

  <div className="fm-diagram">
    <div className="fm-dia-row">
      <div className="fm-dia-card">
        <span className="fm-dia-num">01</span>
        <strong className="fm-dia-label">SCREENSHOT</strong>
        <p className="fm-dia-desc">Captures the viewport so the LLM can see buttons, forms, modals, and overall page layout.</p>
      </div>
      <span className="fm-dia-arrow">→</span>
      <div className="fm-dia-card">
        <span className="fm-dia-num">02</span>
        <strong className="fm-dia-label">EXTRACT DOM</strong>
        <p className="fm-dia-desc">Parses the page into a tree of interactive elements with labels and positions that Playwright can target.</p>
      </div>
      <span className="fm-dia-arrow">→</span>
      <div className="fm-dia-card">
        <span className="fm-dia-num">03</span>
        <strong className="fm-dia-label">LLM DECIDES</strong>
        <p className="fm-dia-desc">Combines the screenshot, DOM, and your prompt to pick a target element and action: click, type, select, scroll, or extract.</p>
      </div>
      <span className="fm-dia-arrow">→</span>
      <div className="fm-dia-card">
        <span className="fm-dia-num">04</span>
        <strong className="fm-dia-label">EXECUTE</strong>
        <p className="fm-dia-desc">Playwright runs the action in real Chromium. Credentials are injected directly into browser fields, never exposed to the LLM.</p>
      </div>
      <span className="fm-dia-arrow">→</span>
      <div className="fm-dia-card">
        <span className="fm-dia-num">05</span>
        <strong className="fm-dia-label">GOAL CHECK</strong>
        <p className="fm-dia-desc">If the goal is met, return results. Otherwise, loop back to step 01 and screenshot the now-changed page.</p>
      </div>
    </div>
    <div className="fm-dia-return">
      <span>↻ repeat until goal is met</span>
    </div>
  </div>

  <h3 className="fm-h3">Planner, Agent, Validator</h3>
  <p className="fm-prose">
    For multi-page tasks, Skyvern 2.0 adds a coordination layer on top of the agent loop. Three components work together so that if something goes wrong partway through, the system detects the failure and retries rather than silently producing incomplete results.
  </p>

  <div className="fm-diagram">

        <strong className="fm-dia-label">TASK AGENT</strong>

      </div>
      <span className="fm-dia-arrow">→</span>

<div className="fm-section" id="concepts">
  <div className="fm-section-head">
    <span className="fm-section-num">03</span>
    <h2 className="fm-section-title">Concepts</h2>
    <p className="fm-section-sub">Eight building blocks for any automation. Select one to explore.</p>
  </div>

<Tabs>
  <Tab title="Tasks">
    A **Task** is a single automation job: you provide a prompt describing what you want done and the URL to start from, and Skyvern navigates the browser and returns structured results.

    The example below extracts the top 3 posts from Hacker News. The `data_extraction_schema` tells Skyvern what shape of data to return, so you get typed JSON instead of freeform text.

    ```python
    from skyvern import Skyvern

    skyvern = Skyvern(api_key="YOUR_API_KEY")

    result = await skyvern.run_task(
        prompt="Find the top 3 posts on the front page",
        url="https://news.ycombinator.com",
        data_extraction_schema={
            "type": "object",
            "properties": {
                "posts": {"type": "array", "items": {"type": "string"}}
            }
        }
    )

    print(result.status)   # "completed"
    print(result.output)
    # {"posts": ["Show HN: AI Browser Agent", "Startup raises $50M", "New open-source framework"]}
    ```

    `prompt` and `url` are the only required fields. Add `data_extraction_schema` to get structured output, `max_steps` to cap cost during development, or `browser_session_id` to run the task inside an existing authenticated browser.

    Use Tasks for one-off automations and data extraction. When you need something reusable with multiple steps, use a Workflow instead.

    <CardGroup cols={2}>
      <Card title="Run a Task (API)" icon="code" href="/running-automations/run-a-task" />
      <Card title="Run a Task (UI)" icon="window-maximize" href="/cloud/getting-started/run-a-task" />
    </CardGroup>
  </Tab>

  <Tab title="Workflows">
    A **Workflow** is a reusable, multi-step template composed of blocks. Unlike Tasks, Workflows can be versioned, shared across your team, and executed repeatedly with different parameters.

    The example below runs a workflow that searches for products. Parameters like `search_query` and `max_price` are passed in at runtime and available to every block inside the workflow via Jinja templates (`{{search_query}}`).

    ```python
    result = await skyvern.run_workflow(
        workflow_id="wpid_abc123",
        parameters={
            "search_query": "wireless headphones",
            "max_price": 100
        }
    )

    print(result.status)   # "completed"
    print(result.output)
    # {"products": [{"name": "Sony WH-1000XM5", "price": 89.99}, {"name": "Bose QC45", "price": 79.00}]}
    ```

    Blocks can also reference outputs from earlier blocks using `{{extract_block.product_name}}`, so data flows naturally through each step of the workflow.

    <CardGroup cols={2}>
      <Card title="Build a Workflow (API)" icon="code" href="/multi-step-automations/build-a-workflow" />
      <Card title="Build a Workflow (UI)" icon="window-maximize" href="/cloud/building-workflows/build-a-workflow" />
    </CardGroup>
  </Tab>

  <Tab title="Blocks">
    **Blocks** are the building units of Workflows. Each block handles one job, and they execute sequentially so each one can reference outputs from the blocks before it. There are 17 block types across four categories.

    #### Navigation and interaction
    AI-guided navigation toward a goal, single targeted actions (click, type, select), direct URL navigation, login with stored credentials, timed pauses, and manual intervention points.

    #### Data and files
    Structured data extraction from pages into JSON, file downloads and uploads, and parsing for PDFs, CSVs, and Excel files.

    #### Logic and control flow
    Conditional branching (if/else), for loops over lists, validation assertions that halt the workflow on failure, and a code block for custom Python or Playwright scripts.

    #### Communication
    Outbound HTTP requests to external APIs, text-only LLM prompts (no browser involved), and sending email.

    <CardGroup cols={2}>
      <Card title="Block Reference (API)" icon="code" href="/multi-step-automations/workflow-blocks-reference" />
      <Card title="Configure Blocks (UI)" icon="window-maximize" href="/cloud/building-workflows/configure-blocks" />
    </CardGroup>
  </Tab>

  <Tab title="Runs">
    Every time you execute a Task or Workflow, Skyvern creates a **Run** to track progress and store outputs.

    **Lifecycle:** `created` → `queued` → `running` → `completed` | `failed` | `timed_out` | `canceled`

    The run response includes everything you need to process results:

    - `run_id` - Unique identifier (`tsk_*` for tasks, `wr_*` for workflows)
    - `output` - Extracted data matching your schema
    - `recording_url` - Full video of the session
    - `screenshot_urls` - Captured after each action
    - `downloaded_files` - Signed URLs with checksums
    - `failure_reason` - Error details if the run failed

    <Warning>You're billed per step. Use `max_steps` to cap costs during development.</Warning>

    <CardGroup cols={2}>
      <Card title="Run Details" icon="chart-line" href="/cloud/viewing-results/run-details" />
      <Card title="Run History" icon="clock" href="/cloud/viewing-results/run-history" />
    </CardGroup>
  </Tab>

  <Tab title="Credentials">
    Credentials store passwords, TOTP codes, and credit cards with encryption at rest and in transit. During execution, values are injected directly into browser fields. The LLM orchestrates the login flow but never sees the actual values.

    **Supported types:** Passwords · TOTP (authenticator apps) · Credit cards

    **Sources:** Skyvern (native) · Bitwarden · 1Password · Azure Key Vault

    Once stored, credentials are referenced by ID in Login blocks or auto-matched when Skyvern detects a login page. The execution flow is: detect the login form, inject username and password into browser fields, then handle 2FA if a TOTP credential is attached.

    <CardGroup cols={2}>
      <Card title="Credentials Overview" icon="lock" href="/cloud/managing-credentials/credentials-overview" />
      <Card title="TOTP Setup" icon="shield" href="/cloud/managing-credentials/totp-setup" />
    </CardGroup>
  </Tab>

  <Tab title="Sessions">
    Skyvern offers two ways to persist browser state: **Sessions** and **Profiles**.

    A **Browser Session** is a live browser instance that stays open across multiple tasks for up to 24 hours. Use sessions when you need to chain tasks in real time, for example logging in with one task and then extracting data with the next while the browser stays authenticated.

    A **Browser Profile** is a saved snapshot of cookies, auth tokens, and local storage. Unlike sessions, profiles persist indefinitely and can be restored on future runs so you don't re-authenticate every time.

    The example below creates a session, runs two tasks inside it (the second inherits the login state from the first), then cleans up.

    ```python
    session = await skyvern.create_browser_session(timeout=120)

    await skyvern.run_task(
        prompt="Log in",
        url="https://example.com/login",
        browser_session_id=session.browser_session_id
    )
    # Browser is now authenticated

    result = await skyvern.run_task(
        prompt="Extract the account balance",
        browser_session_id=session.browser_session_id
    )
    print(result.output)  # {"balance": "$1,234.56"}
    ```

    <CardGroup cols={2}>
      <Card title="Browser Sessions" icon="window" href="/optimization/browser-sessions" />
      <Card title="Browser Profiles" icon="user-circle" href="/optimization/browser-profiles" />
    </CardGroup>
  </Tab>

  <Tab title="Artifacts">
    Every run generates artifacts for debugging, observability, and audit trails.

    - **Recordings** - End-to-end video of the entire run
    - **Screenshots** - Captured after each action for step-by-step review
    - **Downloaded files** - Signed URLs with checksums
    - **Logs** - JSON-structured at step, task, and workflow levels
    - **HAR files** - HTTP Archive data for network-level debugging

    In the Cloud UI, go to **Runs**, click any run, and switch between the **Actions** and **Recording** tabs.

    <CardGroup cols={2}>
      <Card title="Using Artifacts" icon="bug" href="/debugging/using-artifacts" />
      <Card title="Troubleshooting" icon="wrench" href="/debugging/troubleshooting-guide" />
    </CardGroup>
  </Tab>

  <Tab title="Engines">
    Choose which AI model runs your task:

    - `skyvern-2.0` - Latest model, recommended for most use cases
    - `skyvern-1.0` - Previous generation
    - `openai-cua` - OpenAI Computer Use Agent
    - `anthropic-cua` - Anthropic Computer Use Agent
    - `ui-tars` - UI-TARS model

    Set via the `engine` parameter in the SDK or the dropdown in the dashboard.

    ```python
    from skyvern.schemas.runs import RunEngine

    result = await skyvern.run_task(
        prompt="Extract pricing data",
        url="https://example.com/pricing",
        engine=RunEngine.skyvern_v2
    )
    print(result.output)
    # {"monthly": 29, "annual": 290, "enterprise": "Contact sales"}
    ```
  </Tab>
