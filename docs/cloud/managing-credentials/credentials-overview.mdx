The **Credentials** page stores sensitive values — passwords, payment cards, and secrets — so your workflows can use them without embedding secrets in prompts or parameters. Skyvern stores credentials by default — no external service required.

<img src="/images/cloud/credentials-overview.png" alt="Credentials page overview" />

## How Skyvern keeps credentials secure

Credentials are treated as write-once secrets that never reach the LLM, logs, or API responses.

<Steps>
  <Step title="Encrypted at rest">
    When you save a credential, the sensitive data (passwords, card numbers, TOTP secrets) is encrypted and stored in a secure vault — backed by Azure Key Vault under the hood. Only metadata like the credential name and username is stored in Skyvern's database.
  </Step>
  <Step title="Placeholders during execution">
    When a workflow runs, the LLM receives only placeholder IDs like `placeholder_abc123_password`. The AI decides *where* to type on the page, but never sees the real values. No third party — including the LLM provider — ever accesses your actual credentials.
  </Step>
  <Step title="Just-in-time injection">
    At the browser level, the automation layer resolves placeholders to real values and types them directly into the page. After execution, any credential values that appear in response data are automatically masked before storage.
  </Step>
</Steps>

## Quick start

<Steps>
  <Step title="Navigate to the Credentials page">
    Click **Credentials** in the left sidebar under **General**.
  </Step>
  <Step title="Click + Add">
    Choose the credential type: **Password**, **Credit Card**, or **Secret**.
  </Step>
  <Step title="Fill in the details and save">
    Enter the required fields and click **Save**. The credential is immediately available for use in workflows.
  </Step>
</Steps>

### Secret credentials

Secrets store a single sensitive value. Create one from **+ Add → Secret**, give it a name and value, then reference it in any parameter field:

```
{{ credential_name.secret_value }}
```

## Using credentials in workflows

The most common pattern is a **Login block** — select a stored credential from the dropdown and Skyvern handles the entire sign-in flow, including 2FA if configured. See [Block Reference → Login](/cloud/building-workflows/configure-blocks) for details.

For workflows that need different credentials at runtime, add a **Credential parameter** (type: `credential_id`). When someone runs the workflow, they pick which credential to use from a dropdown. See [Workflow Parameters](/cloud/building-workflows/add-parameters) for setup.

---

## External credential providers

If your organization already manages secrets in **Bitwarden**, **1Password**, **Azure Key Vault**, or a **custom vault API**, you can connect them as credential sources instead of duplicating credentials in Skyvern. See [External Credential Providers](/cloud/managing-credentials/external-providers) for setup instructions.

You can mix Skyvern-managed and external credentials within the same workflow.

---

## Deleting credentials

Click the **trash icon** on any credential row. Deletion is permanent — deleted credentials cannot be restored. If a workflow references a deleted credential, it will fail at the login step until you assign a replacement.
